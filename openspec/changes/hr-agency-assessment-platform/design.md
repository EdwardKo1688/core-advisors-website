## Context

台灣人力仲介產業面臨官方評鑑制度升級，從「媒合作業能力」轉向「治理能力」評量。現有系統（SandboxTech Hello）僅提供訪視表管理、雇主/移工 CRUD 等基礎功能，缺乏評鑑指標追蹤、法遵控制、證據鏈管理與 AI 輔助決策。現有雛形網站（React SPA）已具備基礎後台框架（儀表板、AI 診斷、預約管理），可作為 UI 風格參考。

目標用戶為大型人力仲介企業（50-500 人），涵蓋總部與多分公司，需支援 7 種角色同時操作。系統需處理越南、印尼、菲律賓、泰國四國語系的多語內容。

## Goals / Non-Goals

**Goals:**
- 建立以評鑑指標為核心的營運管理平台，讓日常作業自動產生評鑑佐證
- 將法遵規則內嵌到每個作業節點，實現風險前移
- 提供案件全生命週期的統一時間軸，消除資訊孤島
- 透過 AI Copilot 降低文件整理、知識傳承與決策分析的人力成本
- 支援多租戶（多家仲介公司）獨立運作
- 提供跨平台行動端（PWA），取代 Android-only 限制

**Non-Goals:**
- 不取代政府申辦系統（如 EZ Work）的線上申請功能
- 不處理移工薪資發放與會計總帳（與既有 ERP/會計系統整合即可）
- 不建置移工招募端（海外仲介對接）的完整流程
- 第一階段不含即時通訊（LINE/WhatsApp 整合列為後續規劃）
- 不處理移工個人端 APP（移工自助查詢列為後續規劃）

## Decisions

### Decision 1: 單體式應用 vs. 微服務

**選擇：模組化單體（Modular Monolith）**

| 方案 | 優點 | 缺點 |
|------|------|------|
| 微服務 | 獨立部署、技術異構 | 初期複雜度過高、運維成本高 |
| 模組化單體 | 開發效率高、部署簡單、未來可拆分 | 模組間耦合風險 |

**理由**：初期團隊規模 3-5 人，微服務的基礎設施成本過高。模組化單體以清晰的模組邊界（assessment、compliance、case、visit 等）設計，未來需要時可逐步拆分為微服務。

### Decision 2: 前端框架選擇

**選擇：Next.js 14+ (App Router) + TypeScript**

| 方案 | 優點 | 缺點 |
|------|------|------|
| 純 React SPA | 簡單、參考現有雛形 | SEO 弱、首次載入慢 |
| Next.js App Router | SSR/SSG 混合、API Routes 整合、效能佳 | 學習曲線稍高 |

**理由**：Next.js 提供 Server Components 降低 bundle size，API Routes 可整合部分後端邏輯，App Router 的 layout 機制適合多層級後台導航。PWA 支援良好。

### Decision 3: 後端技術棧

**選擇：Node.js + Express + Prisma ORM**

| 方案 | 優點 | 缺點 |
|------|------|------|
| Python/FastAPI | AI 整合方便、資料科學生態 | 前後端語言不一致 |
| Node.js/Express | 全棧 TypeScript、生態成熟 | CPU 密集任務較弱 |

**理由**：與前端共用 TypeScript 生態，Prisma 提供型別安全的 ORM，降低前後端資料模型不一致的風險。AI 相關 CPU 密集任務透過 Worker Thread 或呼叫外部 AI API 處理。

### Decision 4: 資料庫設計策略

**選擇：PostgreSQL + 多租戶 Schema 隔離**

- 每個仲介公司獨立 Schema，共用 `public` schema 存放系統級設定
- 使用 Row-Level Security (RLS) 作為額外安全層
- 評鑑指標使用 JSONB 欄位支援彈性結構（不同年度指標可能調整）
- 文件內容使用 `pgvector` 擴充支援向量搜尋（AI RAG）

### Decision 5: AI 整合架構

**選擇：Claude API + RAG Pipeline**

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  使用者介面   │────▶│  AI Gateway   │────▶│  Claude API  │
│  (Copilot)  │     │  (Router)    │     │             │
└─────────────┘     └──────┬───────┘     └─────────────┘
                           │
                    ┌──────▼───────┐
                    │  RAG Engine  │
                    │  - pgvector  │
                    │  - 法規知識庫 │
                    │  - 案件上下文 │
                    └──────────────┘
```

- AI Gateway 統一管理所有 Copilot 的 prompt routing、rate limiting、usage tracking
- 法規知識庫與評鑑指標透過 Embedding 建立向量索引
- 案件上下文透過 structured context injection 提供（非全文搜尋）
- 支援 streaming response 提升使用者體驗

### Decision 6: 行動端策略

**選擇：PWA (Progressive Web App)**

| 方案 | 優點 | 缺點 |
|------|------|------|
| React Native | 原生體驗、推播方便 | 雙端維護、App Store 審核 |
| PWA | 一套程式碼、免安裝、離線支援 | 推播限制（iOS）、硬體 API 受限 |

**理由**：訪視人員主要需求為填表、簽名、拍照、GPS 定位，PWA 均可支援。免除 App Store 審核流程，可快速迭代。Service Worker 實現離線填表與背景同步。

### Decision 7: 檔案儲存

**選擇：MinIO (S3-compatible) 自建 / AWS S3**

- 契約、訪視照片、簽名檔等透過 presigned URL 直傳
- 檔案 metadata 存入 PostgreSQL，實體檔案存物件儲存
- 支援 CDN 加速（CloudFront / CloudFlare R2）

## Risks / Trade-offs

| 風險 | 影響 | 緩解策略 |
|------|------|---------|
| 評鑑指標每年可能調整 | 資料模型需彈性 | JSONB 欄位 + 版本化指標模板，支援年度更新而不影響歷史資料 |
| 多語內容維護成本高 | 越/印尼/菲/泰/英/中六語 | AI 輔助翻譯 + 範本化訪視內容，減少人工翻譯量 |
| AI API 成本隨用量增長 | 營運成本不可控 | Token 用量追蹤 + 分級使用策略（關鍵場景用 Claude Opus，常規用 Haiku） |
| 仲介公司 IT 能力普遍偏弱 | 系統導入困難 | 漸進式上線（先訪視→再評鑑→再 AI），提供引導式 onboarding |
| 離線同步衝突 | PWA 離線填表後同步可能衝突 | Last-write-wins + 衝突佇列 + 人工審核機制 |
| 個資法（PDPA）合規 | 移工個資敏感度高 | 欄位級加密、存取日誌、資料保留策略、定期清除機制 |
| 現有系統資料遷移 | SandboxTech 資料格式不明 | 提供 Excel/CSV 標準匯入介面，支援批次遷移工具 |

## Open Questions

1. **多租戶定價模型**：按仲介公司規模（移工人數）分級收費？或按功能模組訂閱？
2. **與政府系統整合深度**：是否需要自動帶入 EZ Work 申辦進度？需確認 API 可用性
3. **AI 模型選擇**：Claude 為主，是否需要備援模型（GPT-4o）以降低單一供應商風險？
4. **第一階段範圍優先級**：建議 MVP 順序為 訪視運營 → 評鑑中台 → AI Copilot，需與客戶確認
