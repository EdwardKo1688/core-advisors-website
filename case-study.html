AI 戰情看板導入，讓管理層首次「看到」真實數據

產業：物流 / 電商 規模：中型企業（200–500 人） 服務模式：常駐顧問（3 個月起）
C
Context — 情境與挑戰
顧問介入前的企業現況

該企業為電商平台提供倉配一條龍服務，日均處理訂單量超過萬筆。管理報表仰賴各部門每週人工整理 Excel，彙總後再由主管逐一解讀，從數據產生到決策形成平均延遲 3–5 天。

核心痛點包括：

  • 倉儲效率、車輛 OTD（準時送達率）等關鍵指標無即時追蹤
  • 績效歸因困難，異常出現後才能事後追查
  • 管理層對「AI 能做什麼」有期待，但無法確定從哪個場景切入最有價值
A
Action — 顧問行動步驟
介入方法與執行序列

顧問採三段式導入，以「先求有，再求好」為原則：

  • W1–2 現況診斷：與 IT、倉管、車隊主管進行深度訪談，盤點現有數據源（ERP、TMS、GPS）與可用性，產出 AI 導入適合場景清單
  • W3–4 指標體系設計：依麥肯錫 MECE 原則建立 8 構面 100 項 KPI 體系，按「重要性 × 可取得性」排序，鎖定 Phase 1 的 20 項核心指標
  • W5–8 看板 MVP 上線：協助 IT 完成數據串接，以 BI 工具建立即時戰情看板，管理層可自助查詢每日核心指標；同步進行第一份 Quick Win 報告（導入前 vs. 後的對比）
R
Result — 可量化成效
第 8 週 Quick Win 報告摘要

管理層在第 8 週會議首次基於即時數據進行決策討論；IT 主管表示，看板讓跨部門對話從「說感覺」轉為「看數字」。

E
Extension — 後續延伸
Phase 2 規劃方向

基於 Phase 1 的信任基礎，Phase 2 規劃納入預測性分析（OTD 預測、異常預警）與 AI 排程輔助，預計在 M4–8 實施。企業已核准 Phase 2 預算，進入長期常駐顧問合作模式。

認證審查 AI 化，縮短標章申請等待週期

產業：政府 / 財團法人 規模：小型機構（50–100 人） 服務模式：陪跑顧問（6 個月）
C
Context — 情境與挑戰
顧問介入前的機構現況

該機構負責管理多類建築相關標章認證（含綠建築、智慧建築等），每年受理申請數量持續成長，但審查人力無法同步擴編。申請資料缺件率約 40%，導致大量退件往返，審查人員耗費 60% 以上的時間在行政確認而非專業審查。

公部門文化特殊性：

  • 法規決定流程，AI 只能輔助不能替代最終判斷
  • 資深審查人員對 AI 工具有疑慮，需要漸進式信任建立
  • 預算與採購流程較私部門更長，需提前規劃
A
Action — 顧問行動步驟
公部門節奏的導入設計
  • M1 現況盤點：訪談審查人員與申請人，繪製 13 類標章的現行流程圖，標記人工耗時節點與高缺件類型
  • M2–3 AI 輔助工具設計：設計缺件自動偵測機制（文件類型核對、必填欄位檢查），Human-in-the-loop 架構確保審查員保有最終決策權
  • M4–5 員工 AI 素養培訓：分三層（基礎認知 / 工具操作 / 進階應用）設計培訓課程，先從最有意願的部門試行，建立內部 Champion
  • M6 成效驗證：統計缺件率與退件往返次數，製作導入前後對比報告，向主管機關報告成效

Extension — 後續延伸
方法論可複製性

本案建立的「公部門 AI 導入三原則」（法規優先、Human-in-the-loop、漸進信任）已成為核心顧問服務其他政府機構的標準方法論，可直接複製至類似認證或審查型機構。

Claude Code 導入教練，開發效率提升並建立 AI 平台商模

產業：科技 / SaaS / xIoT 規模：新創（10–50 人） 服務模式：技術顧問教練（持續）
C
Context — 情境與挑戰
顧問介入前的企業現況

該企業提供運輸管理系統（TMS）與車聯網（xIoT）平台服務，技術團隊有一定 AI 基礎，但 AI 輔助開發工具（如 Claude Code）的導入缺乏結構化規範，各工程師使用方式不一致，導致 AI 生成代碼品質參差，code review 成本反而上升。

同時，公司希望從單純 SaaS 工具提供商，轉型為 AI 平台服務商（自有 AI 平台品牌 Rimo），商模轉型缺乏清晰路線圖。

A
Action — 顧問行動步驟
技術賦能 + 商模策略雙軌並行
  • 技術軌 — Claude Code 工作流標準化：引入 SDD（規格驅動開發）方法，每個功能開發前先寫 OpenSpec 規格文件，AI 依規格生成代碼，降低模糊性；建立 Code Review Checklist 與 AI 生成代碼的品質閘門
  • 技術軌 — AI Agent 架構設計：協助設計 Rimo AI Platform 的 Agent 架構（RAG + 多模型 + 權限控管），確保企業級安全需求與彈性擴展性
  • 商模軌 — SaaS → 平台轉型路線圖：分析 TMS 數據資產的核心價值，設計「工具 → 平台 → 生態」三階段商模轉型計畫，並評估 xIoT 車聯網數據作為差異化競爭資產的可行性

Extension — 後續延伸
技術賦能的複利效應

SDD + Claude Code 的工作流已成為該公司的標準開發規範，新加入的工程師可快速上手。Rimo AI Platform 的 MVP 規格已完成，正進入開發階段。顧問持續以月顧問模式協助商模驗證與關鍵技術決策。

IC 通路培訓體系 AI 化，從課程設計到學員互動全面升級

產業：IC 通路 / 知識培訓 規模：中型企業（100–300 人） 服務模式：診斷規劃 + AI 化方案設計
C
Context — 情境與挑戰
顧問介入前的企業現況

某 IC 通路商旗下設有培訓品牌，專門提供半導體產業人才培訓服務。隨著 AI 晶片、車用 IC 等新議題快速演進，課程內容更新週期跟不上市場需求，技術文件整理耗費大量人力，學員在課後常需等待數天才能得到技術問題的回覆。

  • 課程更新：一門課從需求確認到上線平均需要 3 個月
  • 知識管理：技術文件散落各處，講師查找資料效率低
  • 學員互動：課後問答依賴人工，無法即時回應
A
Action — 顧問行動步驟
培訓產業 AI 化路線圖
  • 培訓需求盤點：與課程開發、講師、業務三方訪談，繪製完整培訓生命週期，標記 AI 可介入的 5 個高價值節點
  • AI 輔助課程生成：設計 Prompt 工程框架，讓課程開發人員以 AI 輔助完成講義草稿、案例生成、測驗題設計，縮短內容開發週期
  • 智慧問答知識庫（RAG）：設計以 RAG（檢索增強生成）架構為基礎的技術文件問答系統，讓學員可即時查詢課程相關技術問題，講師不需手動回覆重複問題
  • 培訓 AI 化路線圖：產出 12 個月的分階段 AI 化計畫，含技術選型建議、預算規劃、KPI 設定
